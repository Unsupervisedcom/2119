Project: 2119 Author: T. Willis
Status: In production July 2026

The agent wrote the test.
Who checked the test?

2119 is a small tool that makes plans become requirements, requirements become tests, and tests get judged by someone who didn't write them — enforced, not requested.

1.Why I built this

The most common failure mode in agentic development isn't broken code. It's an agent quietly altering the tests to suit what it built — a tautological assertion here, a mock that bypasses the behavior under test there, a green suite that verifies nothing. The work looks done. The gap only surfaces later, when you're standing on it.

The fix I fell in love with came from DeepWork's DeepReviews: write the plan as an RFC 2119 spec — numbered requirements using MUST, SHOULD, and MAY — then have review agents check that every requirement has a genuine test. Once I started running projects this way, the difference wasn't subtle:

I cannot tell you the extent to which that is magic secret sauce to things just working. All the stuff that I have runs on that, and it is pretty damn good at making it shockingly solid. — a teammate, describing the same pattern on our weekly call

It's a load-bearing concept that's also easy to hold: the words you use to write a plan are the words that determine how it gets tested. 2119 exists to make that one idea trivially adoptable — one command, any repo, any coding agent, no larger framework required.

2.The loop

My build workflow now has four steps. The first is unchanged from how everyone works; the other three are what 2119 enforces.

CONTEXT

Prompt the agent to read the relevant context and get prepared to discuss the feature or product idea.

SPEC

Have it plan first, and store the plan as an RFC 2119 spec: numbered requirements with stable IDs, exactly one keyword each. 2119 lint enforces the format.

BUILD

Have it implement the spec. Every MUST needs at least one test carrying that requirement's ID in a comment. 2119 cover fails on any gap, in either direction.

JUDGE

A fresh-context reviewer — one that didn't write the code — reads each requirement and its tests and answers one question: would these tests fail if this requirement were violated? Verdicts are recorded, committed, and gate 2119 check.

A spec looks like this — plain markdown, every line addressable:

# REQ-001: Session Handling

### REQ-001.1: Timeouts

1. Sessions MUST expire after 30 minutes of inactivity.
2. Expired sessions MUST NOT be resumable with a stale token.
3. The docs SHOULD explain the timeout rationale. [review: docs/**]

And a covering test declares which requirement it exists for — a comment, so it works in any language:

// 2119: REQ-001.1.2
it("rejects a stale token after expiry", async () => {
  ...
});

3.Why it's hard to cheat

Each layer catches exactly the thing the layer before it can't see:

LayerQuestion it answersWhat it catches
lintIs the plan actually a spec?Vague plans, keyword soup, unstable IDs
coverDoes every requirement have a test?Untested MUSTs, tests pointing at dead requirements
reviewWould the test fail if the requirement broke?Tautologies, over-mocking, keyword theater
checkAll of the above, one exit codeRuns identically in hooks, pre-commit, and CI

The part that makes review verdicts trustworthy is a content hash. Every verdict is keyed to a SHA-256 of the requirement's text plus the full content of its covering tests:

requirement text ─┐
                  ├─▶ sha256 ─▶ REQ-001.1.2--9f2c41ab07de ─▶ verdict: pass
covering tests  ──┘

  ...then the agent edits a covering test...

requirement text ─┐
                  ├─▶ sha256 ─▶ REQ-001.1.2--e07d5512c9aa ─▶ no verdict — check fails
covering tests* ──┘

Edit a test — or reword the requirement — and the old approval silently stops counting. There is no way to reuse a stale pass, and 2119 pass refuses any verdict whose hash doesn't match current content, so approvals can't be pre-computed or replayed. Verdicts are committed JSON with a written justification, which means every review decision is visible in the PR diff for humans to audit.

Stated honestly: no local tool can physically stop an agent from approving its own work, because the agent controls the shell. The mitigations are layered — committed verdicts make self-approval visible, hash invalidation makes it short-lived, instruction files direct dispatch to a fresh-context subagent, and CI re-runs the same check so nothing merges without the full gate.

It works on its author, which is the only evidence that matters. Building 2119 with 2119, the first review round returned six failing verdicts — every one a genuine defect, including a masked assertion, a coverage test that only exercised one of three legs, and the parser violating the tool's own spec. The reviewers were right all six times.

4.Why not just prompt harder?

Because instructions decay and incentives don't. An agent under pressure to finish will satisfy the letter of "write tests for everything" with tests that can't fail. Asking it to also police itself is asking the same incentive to hold the whistle.

The design bet in 2119 is that determinism should carry the enforcement weight. Lint and coverage are exact parsing — an agent can't talk its way past an exit code, and CI runs the same command with no agent in the loop at all. Judgment is reserved for the one question that genuinely needs it (is this test honest?), and even that judgment gets pinned to content hashes the moment it's rendered. Hooks make the feedback fast; they were never the wall.

The other thing you get is a golden thread. A claim in the plan traces to a numbered requirement, the requirement traces to named tests, the tests trace to a judged verdict. When someone asks "does it actually do what we said?", the answer is a command, not a vibe.

5.Using it

npx rfc2119 init   # specs/, config, AGENTS.md section

On agents with lifecycle hooks — Claude Code, Codex CLI, Gemini CLI — 2119 init --agent <name> adds write-time lint feedback and a stop gate that blocks "done" while check fails. Everywhere else, the AGENTS.md section, a git pre-commit hook (--git-hook), and CI (--ci) enforce the same gate with slower feedback. Same command, same verdicts, any agent.

It composes with DeepWork rather than replacing it: DeepReviews remains the general engine for diff-triggered custom review rules; 2119 is the portable distillation of one specific loop — the one I now start every project with.
